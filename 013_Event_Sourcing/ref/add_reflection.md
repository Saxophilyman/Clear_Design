Разбиваем программу на события и команды.
События все хранятся в отдельной "истории" (EventStore).
Состояния "восстанавливаются" из EventStore

Основные блоки:
 - Command  
 - Event
 - EventStore
 - CommandHandler

Command решает, какие события должны появиться.  
Event знает, как изменить состояние.  
EventStore только хранит события.  
CommandHandler только координирует процесс.  

----

Мой вариант:
CommandHandler
 - получает command
 - читает events из EventStore
 - восстанавливает state
 - просит command.decide(state)
 - сохраняет новые events

пересборка состояния:
 - events -> event.apply(state) -> final_state

Ещё немного о варианте:  
Событие (Events) — это факт, который уже произошёл. 
Робот что-то **сделал** - и через apply применяет это событие к состоянию.  
Command - намерение к действию, но не действие. Может быть отклонена.
EventStore - хранилище событий в памяти. append - добавляет список новых событий в общую историю.
robot_id - позволяет работать с конкретным роботом.  
```python
def rebuild_state(events: list[RobotEvent]) -> pure_robot.RobotState:
    state = create_initial_state()

    for event in events:
        state = event.apply(state)

    return state
```
Восстановление состояния(Time Travel ниже) - rebuild_state. Состояние не хранится как главное.
Главное — история событий. Чтобы узнать текущее состояние робота, берётся начальное состояние:
`state = create_initial_state()` и последовательно применяются все события:
```
for event in events:
    state = event.apply(state)
   ```
CommandHandler - получает хранилище событий. Он нужен, чтобы обработать команду по полному сценарию:
`получить историю -> восстановить состояние -> принять решение -> сохранить события`
```python
def handle(self, command: RobotCommand) -> list[RobotEvent]:
    pass_events = self.event_store.get_events(command.robot_id) # берём все прошлые события робота по id.
    current_state = rebuild_state(pass_events) # восстанавливаем текущее состояние робота.
    new_events = command.decide(current_state) # команда смотрит на текущее состояние и решает, какие события должны произойти
    self.event_store.append(new_events) # новые события сохраняются в историю.
    return new_events
```

Сама схема:
```
Command
|
v
CommandHandler
|
v
загрузить прошлые события
|
v
rebuild_state(events)
|
v
current_state
|
v
command.decide(current_state)
|
v
new_events
|
v
event_store.append(new_events)
```
Примечание:
пересобирать все события каждый раз может быть дорого, в реальности добавляют:   
snapshots;  
кэш текущего состояния;  
сохранение последней версии состояния;  
индексы событий;  
ограничение глубины восстановления. 

Event Sourcing не стирает прошлое.
Он хранит прошлое и позволяет строить состояние на его основе.

---------
Исследование эталона и ключевые отличия

Повторение:  
**События (Events)** - неизменяемые объекты, описывающие что произошло  
**Event Store** - хранилище всех событий, единственное мутабельное место  
**State Projector** восстанавливает состояние из событий  
**Command Handler** обрабатывает команды, восстанавливая состояние перед каждой операцией  
**Time Travel** позволяет получить состояние на любой момент времени  

Плюсы/минусы указаны в материале

id робота отвязаны от команд  
класс StateProjector вместо rebuild_state - получаем отдельный модуль, который строит состояние из событий
TimeTravel - отдельный логический модуль для восстановления состояния
EventStore хранит события по роботам(их id), у меня - всё в списке для фильтрации

В целом гораздо более чёткое разделение ответственности и архитектурных ролей
